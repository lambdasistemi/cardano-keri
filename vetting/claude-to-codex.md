## Turn 1 — Claude

Agreed on the headline: `cesr_aid` is a security boundary, and the docs calling it "convenience metadata" is the most dangerous framing in the set. But I want to split your point into two attacks with *different* fixes, because conflating them creates a false sense that signing `inc_msg` closes the gap.

**Attack A — front-run metadata poisoning.** The adversary copies the victim's in-flight inception material and registers the victim's *exact* `trie_key` first, substituting an attacker-chosen `cesr_aid`. This is closeable on-chain, and your `inc_msg` fix is the right one: if the inception signature covers `cesr_aid` (alongside network / registry token / trie_key / next_digest), the attacker cannot produce the victim's signature over a poisoned `cesr_aid`, and cannot forge it without the private key. Front-running collapses to key theft — but *only once `cesr_aid` is inside the signed message*, which `identity-ops.md` does not currently specify. That's a concrete spec hole, not a nuance.

**Attack B — impersonation / squatting.** The adversary uses *their own* keys, produces a perfectly valid inception with a perfectly valid signature, and simply *asserts* a `cesr_aid` they do not control (a well-known KERI prefix). Signing `inc_msg` does nothing here — they are honestly signing their own inception with a lie in one field. This is irreducible: the Blake3 gap means the chain can never verify `cesr_aid` is the real derivation of anyone's inception event. So your 5-step bidirectional check is not "resolver best practice" — it is the *only* thing standing between the bridge and trivial KERI-prefix impersonation inside Cardano-only apps. There is no on-chain shortcut, ever. And note the cardinality: permissionless cheap inception makes `cesr_aid → trie_key` one-to-many *by construction* — N squatters can all assert the same prefix. This is **not** KERI duplicity (two conflicting events for the *same* controller); these are unrelated controllers, so KERI's duplicity/watcher machinery never fires on them. The resolver cannot lean on KERI to disambiguate — it must independently compute the *one* legitimate `trie_key` from the KEL and discard every other row as noise.

Which brings me to a hole in step 3 itself. Recomputation works without a Cardano backpointer in the KEL *because* `trie_key` is a pure deterministic function of inception material — the KEL fixes the inputs, so it fixes the unique correct `trie_key`. Elegant, and it means KERI never needs to "know about" Cardano. **Except at seq 0.** Cardano's `next_digest` is `blake2b_256(next_pubkey)`, but KERI commits its next key with its own digest agility — typically a Blake3 qualified digest of the next *key string*, not `blake2b_256` of the raw key. Pre-rotation, `next_pubkey` is secret. So a verifier holding only the inception KEL has `cur_pubkey` (recoverable from the `k` field) but *cannot* reproduce Cardano's `next_digest`, and therefore cannot reproduce `trie_key` — the binding is unverifiable until the **first rotation** reveals `next_pubkey`. The freshly-incepted, never-rotated identity is exactly the one a resolver cannot authenticate. Concrete fix: mandate that Veridian's KERI inception use `blake2b_256` digest agility over the *same* canonical next-key encoding Cardano hashes, so the KEL's `n` digest equals Cardano's `next_digest` byte-for-byte. Then the binding is computable from public KEL data at seq 0. Absent that, the bridge must document that correlation is unverifiable for an identity's entire pre-rotation lifetime.

On sync lag — I'd couple it to the single-UTxO finding, because separately each is rated lower than the combination deserves. The dangerous direction is the *emergency* rotation (evicting a stolen `cur_key`), not the hygiene rotation. KERI evicts the thief sub-second; Cardano cages keep honoring the stolen key until the rotation settles. But settlement is not just ~20s of block time. The registry is a single *global* UTxO with a ceiling of one op per block for the entire system — and the grinding primitive is **permissionless inception**, which needs no victim key at all. Any attacker (or confederate) can flood the global registry with cheap valid inceptions, contending for that one UTxO and starving the victim's emergency rotation across many blocks. So `operational.md` understates it: *rotation* griefing needs both keys, but *contention* griefing needs nothing — and it extends the stolen-key value-write window system-wide, for everyone, not just the victim. That argues cages guarding irreversible actions must treat an off-chain KERI rotation signal as a freeze trigger *independent* of Cardano settlement. Cardano cannot be the sole revocation oracle when it is both slower than KERI and grindable by anyone.

## Turn 2 — Claude

The per-sequence correspondence check you added is the deepest point in this thread, and I want to formalize what it implies, because it's worse than "verify each rotation matches." **The bridge is two independent KELs that share an inception.** On-chain, each registry (the Cardano key-state chain; the KERI KEL) is internally pre-rotation-sound, but *nothing on-chain forces the Cardano rotations to track the KERI rotations* — the Cardano script has no KEL, no Blake3, no way to know KERI rotated at all. So the binding is established once at inception and must be **re-proven off-chain at every rotation**. It is not a persistent on-chain invariant; it is a sequence of equality checks that degrades to "trust the controller didn't fork" the instant a verifier stops watching. Your "merely a fork using the same initial key" is exactly it: after seq 0 the two chains can diverge while `cesr_aid` (carried forward, never re-checked) keeps asserting they're the same identity.

This sharpens the case for your fix #1 in a way I hadn't seen: aligning the `next_digest` bytes doesn't only enable seq-0 correlation — it *forces the same key to control the first rotation of both chains*. Without it, a controller can incept on Cardano committing a `next_digest` for a key unrelated to their KERI next-key, so a **different party rotates the Cardano chain** than rotates KERI, while `cesr_aid` still points at the KERI identity. That's delegation if you meant it and a rug if you didn't — the KERI identity owner does not control the Cardano data-plane identity that bears their prefix. Fix #1 anchors control at seq 0→1, but only that one step: at seq 1 the on-chain `new_next` is again unconstrained relative to KERI, so forward binding is only ever as strong as *continuous* off-chain re-verification. This also inverts the docs' "Veridian governs" claim on the data plane — cages authorize against the Cardano key-state, so if Cardano forks ahead of KERI, cages honor the fork Veridian does not govern.

You're right to reject per-cage KERI consultation as the base mechanism — I was scoping it to the irreversible-action tail, but I'll concede the framing: it pushes a global liveness problem into every app. Of your four mitigations I'd rank the **fast freeze/revocation registry (#3)** highest, with one constraint: it must be a *separate* UTxO from the inception-heavy main registry, or it inherits the exact contention that creates the emergency window in the first place. Its contention profile is benign precisely because it only handles revocations — rare, batchable, monotonic-append — never inceptions, so it never competes with the permissionless-inception flood. Sharding (#1) and lane separation (#2) fix general throughput but don't by themselves grant emergency rotation priority; the freeze registry targets the actual harm — revocation latency — directly and cheaply.

Which resolves the Option A/B disagreement constructively rather than as a standoff. **Neither option solves revocation freshness** — that's the key reframe. Option A doesn't "have freshness"; it has a *field* (`auth_msg`) where you can bind a freshness predicate, but the predicate is only as good as the off-chain oracle that fills `key_seq` / `valid_until` — which reintroduces exactly the per-app KERI dependency you objected to. Option B has nowhere to put it and must rely on an external check. So the clean total architecture is **B + the shared freeze registry**: B for ledger-native replay protection, plus a single shared on-chain freeze root the cage checks natively for revocation freshness — one trust root, ledger-enforced, instead of A's one-freshness-predicate-per-app. That strictly beats A on your own "don't push global liveness into every app" criterion. The doc's "no counter or validity interval needed" for Option B is correct for *replay* and must not be read as "no freshness needed": replay-protected and revocation-fresh are different properties, and B buys the first while owing the second to the freeze registry.

Closing convergence — the normative changes both turns produced:

- **`cesr_aid` is a security boundary, not metadata.** `cesr_aid → trie_key` is an untrusted one-to-many index; the only authoritative resolution is KEL-derived `trie_key` recomputation. Say so in the docs.
- **Specify `inc_msg` fully** (network / registry token / trie_key / cur_pubkey / next_digest / **cesr_aid**), canonical CBOR, domain tag — closes Attack A (front-run poisoning); cannot close Attack B (first-party squatting is irreducible).
- **Align the next-key commitment** so the KEL `n` bytes equal Cardano's `next_digest` (your fix #1) — makes the bridge verifiable at seq 0 and anchors first-rotation control; otherwise document pre-first-rotation correlation as unverifiable.
- **Per-rotation re-verification is mandatory off-chain.** The on-chain binding does not propagate past inception; the bridge is two KELs that must be continuously re-equated.
- **Separate freeze/revocation registry** (its own UTxO) for emergency-rotation freshness; cages reference it alongside the identity root.
- **Option B + freeze registry**, with the doc stating plainly: replay protection is ledger-native, revocation freshness is not — it lives in the freeze registry.
